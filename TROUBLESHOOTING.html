01

Quick health check

Run these five commands in order whenever the UI is unreachable. Each one isolates a specific layer of the stack.

  • 01
    All containers running?
    docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
    expect: 11 Up
  • 02
    nginx config valid?
    docker exec aeronet-nginx nginx -t
    expect: ok
  • 03
    nginx responding internally?
    curl -sk https://127.0.0.1 -o /dev/null -w "%{http_code}"
    expect: 200
  • 04
    NAT pointing to correct container IP?
    sudo nft list table ip nat | grep dnat — compare IP with docker inspect aeronet-nginx --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'
    must match
  • 05
    Ports 80/443 bound on host?
    ss -tlnp | grep -E ':(80|443)'
    expect: 0.0.0.0
ℹ
UFW is inactive by design

Firewall rules are managed entirely by nftables. sudo ufw status will always show inactive — this is correct and expected.

02

Incident: 2026-03-28 — UI unreachable

Symptom: Browser showed ERR_CONNECTION_REFUSED on http://aeronet.local. Stack had been working the previous day from the same network.

Resolution time: ~2 hours of diagnostics. Resolved by regenerating the TLS certificate and performing a full compose down/up to flush stale Docker NAT rules.

✕
Initial error

All 11 containers were running and ports 80/443 were bound to 0.0.0.0. nginx returned HTTP 200 on its container IP directly, but curl https://216.152.167.28 returned 000 (TCP refused).

03

Root cause analysis

Two separate issues compounded to produce the outage. They occurred in this order:

2026-03-27 · ~19:00
TLS certificate was missing / corrupted
An attempt to run openssl inside the nginx container failed (nginx:1.27-alpine has no openssl in PATH). The certificate files at infra/nginx/certs/ were absent or invalid on this new server deployment, causing nginx to crash on startup. The previous session's nginx was still running from 9 days earlier with the old cert in memory.
2026-03-27 · 19:29
journalctl evidence
OCI runtime exec failed: exec: "openssl": executable file not found in $PATH — confirmed the cert generation attempt ran inside the container rather than on the host.
2026-03-28 · 13:04
docker restart caused stale NAT entry
When docker restart aeronet-nginx was used (instead of compose down/up), the container got a new internal IP (172.18.0.5) but Docker's iptables-nft NAT rules still pointed to the old IP (172.18.0.11). Inbound traffic was DNAT'd to a non-existent address and silently dropped.
2026-03-28 · 13:10
Cert regenerated on host (correct)
New cert generated directly on the host OS using openssl req, with correct separate -keyout aeronet.key and -out aeronet.crt paths. RSA key verified, SAN covers *.aeronet.local and IP:216.152.167.28.
2026-03-28 · 13:20
Full compose down/up flushed stale NAT
docker compose down && docker compose up -d tore down the entire network, reassigned IPs, and rebuilt all iptables-nft NAT rules correctly. nginx was assigned 172.18.0.11 and the DNAT rules matched. Stack fully restored.
04

Fix: TLS certificate regeneration

⚠
Always run openssl on the host, never inside the nginx container

nginx:1.27-alpine does not have openssl in its PATH. Running docker exec aeronet-nginx openssl ... will fail silently or corrupt existing files.

Step 1 — Generate cert on the host

bash — run on Virgilio host
# Navigate to the certs directory
cd /opt/aeronet-os/infra/nginx/certs

# Generate key and cert as SEPARATE files
sudo openssl req -x509 -newkey rsa:4096 -sha256 -days 3650 -nodes \
  -keyout aeronet.key \
  -out aeronet.crt \
  -subj "/CN=aeronet.local" \
  -addext "subjectAltName=DNS:aeronet.local,DNS:*.aeronet.local,IP:216.152.167.28"

Step 2 — Verify both files

bash
# Key must be valid RSA
openssl rsa -in aeronet.key -check -noout
# Expected output: RSA key ok

# Cert must show CN=aeronet.local and correct dates
openssl x509 -in aeronet.crt -noout -subject -dates

Step 3 — Restart the full stack (not just nginx)

bash
cd /opt/aeronet-os
docker compose -f infra/docker-compose.yml --env-file .env.secret down
docker compose -f infra/docker-compose.yml --env-file .env.secret up -d
✕
Do not use docker restart aeronet-nginx after cert changes

A bare docker restart bypasses Docker Compose's network management. The container may get a new internal IP while the NAT table retains the old one, causing a silent routing failure even though nginx appears healthy.

05

Fix: Stale Docker NAT rules

If curl -sk https://127.0.0.1 returns 200 from inside Virgilio but connections from outside are refused, the Docker NAT table likely has a stale container IP.

Diagnose

bash
# Get current nginx container IP
docker inspect aeronet-nginx \
  --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'

# Get what NAT is actually routing to
sudo nft list table ip nat | grep dnat

# These two IPs must match — if they don't, NAT is stale

Fix — full compose cycle

bash
cd /opt/aeronet-os
docker compose -f infra/docker-compose.yml --env-file .env.secret down
docker compose -f infra/docker-compose.yml --env-file .env.secret up -d

# Verify NAT now matches container IP
sudo nft list table ip nat | grep dnat
docker inspect aeronet-nginx \
  --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'

Nuclear option — restart Docker daemon

If compose down/up does not flush the stale rules (rare), restart the Docker daemon to force a complete NAT rebuild:

bash
sudo systemctl restart docker
cd /opt/aeronet-os
docker compose -f infra/docker-compose.yml --env-file .env.secret up -d
06

Fix: Full stack restart procedure

Use this procedure for any restart, not just after cert changes. It is the only safe way to restart — bare docker restart on individual containers is discouraged.

bash — canonical restart procedure
cd /opt/aeronet-os

# Tear everything down cleanly
docker compose -f infra/docker-compose.yml --env-file .env.secret down

# Bring back up — postgres healthcheck gates dependent services
docker compose -f infra/docker-compose.yml --env-file .env.secret up -d

# Confirm all 11 containers are Up
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'

# Tail logs for 30s to catch early failures
docker compose -f infra/docker-compose.yml logs -f --tail=20
⚠
.env.secret must be present

Running compose without --env-file .env.secret will start containers with blank passwords and keys. The file lives at /opt/aeronet-os/.env.secret and must not be deleted or moved.

08

Known issues registry

CRITICAL TLS cert missing on new server deployments ▸

The cert files are not committed to git (correctly — private keys must never be in source control). On any fresh clone or new server, infra/nginx/certs/ will be empty. nginx will refuse to start. Always run the cert generation step in §04 immediately after cloning on a new host.

Workaround: Add a generate-certs.sh script to the repo that runs the openssl command, to be executed once post-clone.

CRITICAL Stale Docker NAT after bare docker restart ▸

Using docker restart <container> directly (instead of compose) can reassign container IPs while leaving the iptables-nft DNAT rules pointing to old IPs. Symptoms: nginx reports healthy internally, external connections silently refused. Fix: always use compose down/up. See §05.

HIGH compose restart warns about missing env vars ▸

Running docker compose restart <service> without --env-file .env.secret prints warnings like "POSTGRES_PASSWORD variable is not set" and also fails to find service names when the compose file uses variable interpolation in service names. Always pass --env-file .env.secret, or use docker restart <container-name> for single-container restarts when env vars aren't needed (e.g. nginx config reload).

HIGH Grafana stat panels show "No data" ▸

Expected until devices are added to the inventory. PostgreSQL-backed panels require rows in the devices table. Prometheus panels work immediately. No fix needed — use the Discovery page to scan and register devices.

MEDIUM MIST collector POLL_ERROR / ADAPTIVE_SLOWDOWN ▸

Expected behaviour when MIST_API_TOKEN and MIST_SITE_ID are placeholder values in .env.secret. The collector uses exponential backoff so these warnings are harmless. Resolve by populating real Juniper MIST credentials.

MEDIUM Server is being actively scanned by internet bots ▸

With ports 80/443 exposed directly to the internet (Firezone VPN not yet deployed), nginx logs will show constant probe traffic from Censys, Palo Alto Xpanse, and various exploit scanners. This is normal for any internet-exposed IP. The React app returns HTTP 200 for all paths (SPA fallback), so scanners see a response but gain nothing. Priority: implement Phase 3 Firezone WireGuard VPN to lock down 80/443.

Security notes

✓
Active hardening measures

SSH on port 2222 (key-only), nftables default-drop, fail2ban (3 attempts / 1hr ban), auditd ISO 27001 rules, no inter-VLAN routing, IP forwarding disabled, all secrets Fernet-encrypted at rest.

fail2ban — check and unban

bash
# Check currently banned IPs
sudo fail2ban-client status sshd

# Unban a specific IP (e.g. your own after lockout)
sudo fail2ban-client set sshd unbanip <YOUR_IP>

# View nftables banned set directly
sudo nft list set inet f2b-table addr-set-sshd

nftables — verify rules intact

bash
# Full ruleset — confirm input chain has default drop policy
sudo nft list table inet filter

# Expected input chain policy: drop
# Expected: port 2222, 80, 443 accepted on OUTSIDE interface
# Expected: INSIDE.4 fully accepted (management VLAN)
⚠
Pending: Phase 3 — Firezone WireGuard VPN

Ports 80 and 443 are currently exposed directly to the internet. Once Firezone is deployed, these should be locked to VPN clients only, eliminating the bot scan surface entirely.
